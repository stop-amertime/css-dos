# Raw player: identical-to-calcite chrome + paintable CSS pixel grid

Date: 2026-06-30
Status: design (approved pending user spec review)

## Problem

`web/player/raw.html` - the "theoretical" pure-CSS player - is broken
in two ways:

1. **It doesn't look like the real player.** It has none of
   `calcite.html`'s chrome (no menu bar, status line, EDIT-style
   window, dialog shadow) and ships a *different, incomplete*
   keyboard: a `<key-board>` element with only ~30 keys, missing
   Caps/Ctrl/Shift/Del/F1–F10/comma/period/slash, and laid out
   differently from the player the user actually sees.

2. **Its 64,000-element pixel grid is inert.** No cabinet emits any
   pixel-painting CSS. The Mode 13h framebuffer is just guest memory
   at `0xA0000`; in the calcite/bridge path the *renderer* turns those
   bytes into an `<img>`. Pure-CSS video output - cabinet CSS lighting
   up DOM pixels - has never existed. So even if Chrome survived
   parsing, the grid would stay black.

The goal: raw.html should be **pixel-identical to calcite.html except
that the `<img id="calcite-screen">` is replaced by the 64,000-element
CSS grid**, it should load `/cabinet.css` and try to run, and the grid
must be **genuinely paintable** by spec-compliant CSS (it will hang/
crash Chrome at full size - that is the whole point of the "raw"
player; correctness, not survival, is the bar).

## Architecture findings (verified against code)

- **Screen pixels are not currently emitted by anything.** Built
  `doom8088.css` has only `.clock`, `.cpu`, `.cpu::after` rule heads
  and `:has(#kb-*:active)` keyboard rules - zero `#pN` / per-pixel
  rules. Confirmed by grepping the built cabinet.
- **Framebuffer = memory.** `kiln/memory.mjs`: Mode 13h framebuffer is
  the zone `[0xA0000, 0xAFA00]`, emitted only when `prune.gfx ===
  false` (opt-in via builder `memory.gfx`). The comment is explicit:
  bytes are "consumed by calcite's framebuffer renderer."
- **Memory storage shape.** Bytes are packed `PACK_SIZE=2` per cell
  into `@property --mc{cellIdx}` integers on `.cpu`
  (`cellIdxOf(addr)=addr>>1`, `cellOffOf(addr)=addr&1`). Byte
  extraction is `mod(var(--mc{idx}),256)` for offset 0,
  `round(down, var(--mc{idx})/256)` for offset 1 - the exact idiom
  already used in `emit-css.mjs` for the LBA register.
- **DAC palette is always emitted.** `[DAC_LINEAR, DAC_LINEAR+768]`
  (`DAC_LINEAR=0x100000`) is in the standard zone set unconditionally.
  256 entries × 3 bytes (R,G,B), stored as 6-bit 0..63 values; the
  renderer does the 6→8-bit expansion.
- **Keyboard contract.** `kiln/template.mjs` `KEYBOARD_KEYS` +
  `emitKeyboardRules()` emit `.cpu:has(#kb-X:active){--keyboard:...}`.
  The player just needs DOM elements with those exact `#kb-X` ids,
  inside `.cpu`. calcite.html already has the full, correct set.

## Design

### Part A - raw.html chrome identical to calcite.html

`raw.html` is autogenerated by `web/scripts/raw-regen.mjs`. Rewrite
that generator so its output is `calcite.html` **byte-for-byte except**:

- The `<img id="calcite-screen" src="/_stream/fb">` inside
  `.screen-cell` is replaced by the pixel grid container
  `<div class="screen-grid">#p0..#p63999</div>` (320×200 `<i>`
  elements).
- The bottom status-line label reads `RAW` instead of `CALCITE`, and
  the `<title>`/menu reflect the raw player. (Cosmetic, keeps the two
  distinguishable; everything else identical.)
- The cabinet is loaded as a **stylesheet** (`<link rel="stylesheet"
  href="/cabinet.css">`) rather than the `<link rel="alternate">`
  calcite.html uses - raw.html's whole purpose is to let Chrome
  actually apply the cabinet.

To keep the two files honestly in sync, the generator reads
`calcite.html` at regen time as the template, performs the two/three
documented substitutions, and writes `raw.html`. The keyboard,
window, menu bar, fonts, and all CSS come from calcite.html verbatim,
so they can never drift. The grid keeps the existing scaffolding CSS
(`display:grid; 320×1px columns; image-rendering:pixelated`) sized to
sit where the `<img>` did inside `.screen-cell`.

`#kb-*` ids already match (calcite.html has the full set); they must
remain descendants of an element carrying `.cpu` so the cabinet's
`:has()` rules fire. calcite.html's `.cpu`/`.clock` wrapper
expectations: confirm during implementation that the cabinet's
`.cpu`-scoped selectors have a `.cpu` ancestor in the player DOM; add
the `.clock > .cpu` wrapper around screen+keyboard if calcite.html
relies on the bridge to supply it. (calcite.html itself has no `.cpu`
node - the bridge renders into `<img>` and never needs one. raw.html
*does* need `.clock`/`.cpu` wrappers so cabinet rules have a host.)

### Part B - paintable pixel grid (new Kiln emit path)

Add a new emitter, `emitPixelPaintRules(opts)` (new file
`kiln/pixels.mjs`, wired into `emit-css.mjs`), that emits one CSS rule
per Mode 13h pixel. **Always emitted** (per decision), independent of
whether gfx memory is pruned - if the framebuffer cell for a pixel
isn't in the address set, the pixel reads its `--mc` initial value
(0 = palette entry 0 = black), which is correct.

For pixel `i` (0..63999), linear framebuffer address `A = 0xA0000+i`:

1. Colour index `ci` = byte at `A`, extracted from
   `--mc{A>>1}` via the offset-0/1 idiom above.
2. DAC lookup: R = byte at `DAC_LINEAR + ci*3 + 0`, G = `+1`,
   B = `+2`. But `ci` is a *runtime* value, so the rule can't pick a
   fixed DAC address. Two viable encodings - pick during
   implementation, prove on the small grid:
   - **(B1) Indexed via custom property + `style()` cascade.** Emit
     the pixel's colour index into a per-pixel custom property, then a
     256-arm `style()` cascade (shared, not per-pixel) maps index →
     `rgb()` from the live DAC cells. Cost: 64k tiny rules + one 256-
     arm palette rule. This keeps the palette *live* (DAC writes
     update colours), matching Chrome semantics.
   - **(B2) Per-pixel inline DAC arithmetic.** Each pixel rule inlines
     the full index→DAC→rgb computation. Simpler to reason about but
     64k× larger and repeats the 256-way lookup per pixel.
   Recommendation: **B1** - far smaller, and "live DAC" is the
   spec-correct behaviour. The 256-arm palette map is written once.
3. Paint: `#p{i} { background-color: rgb(R6*255/63, G6*255/63,
   B6*255/63); }` (6→8-bit expansion, matching the renderer).

The pixel `<i>` elements live in raw.html's grid (Part A). In the
calcite/bridge players these rules are **inert** - calcite renders the
framebuffer to `<img>` and never consults DOM `background-color`, and
those players have no `#pN` elements - so correctness of the existing
path is untouched. The only cost to non-raw cabinets is size/compile
time (~64k rules). This is an accepted tradeoff (decision: always
emit); the spec notes it as the one downside and the natural future
escape hatch (a build flag) if compile budget regresses unacceptably.

### Cardinal-rule check

The pixel emitter reasons only about CSS shape: "for each framebuffer
address, read its memory cell, index a palette, set a colour." It
encodes Mode 13h geometry (320×200, DAC at a fixed shadow address) -
which is **CSS-DOS platform knowledge living in Kiln, the CSS
*emitter***, exactly where VGA/DAC knowledge already lives
(`memory.mjs`, `patterns/misc.mjs`). It does **not** live in calcite.
Calcite still sees only `--mc` integer cells and `background-color`
rules; nothing about x86/DOS/Doom enters the engine. The cardinal rule
("calcite knows nothing above CSS") is not touched - this is emitter
work, and the emitter is *allowed* to know the platform.

## Verification

Per decision - **small-grid proof + reasoning**, not full 320×200
(which hangs Chrome).

1. **Unit (generator):** `raw-regen.mjs` output diffed against
   `calcite.html` shows only the documented substitutions (grid for
   img, RAW label, stylesheet link, `.cpu` wrapper). Assert the grid
   has exactly 64,000 `<i id=pN>` and the full `#kb-*` id set is
   present.
2. **Pixel-paint correctness on a tiny grid.** Add a test mode to the
   pixel emitter (or a standalone fixture) that emits an N×M grid
   small enough for Chrome (e.g. 8×8 or 32×20). Build a fixture
   cabinet whose framebuffer + DAC are seeded to known values, render
   in real Chrome via Playwright, read back each pixel's computed
   `background-color`, and assert it equals the expected
   index→DAC→rgb colour. This proves the CSS arithmetic is
   spec-correct. Run it through calcite too (calcite must produce the
   same `--mc`/colour values) to keep the cardinal rule honest.
3. **Scale + reason.** With the arithmetic proven on the small grid,
   scale the generator to 320×200. Document that the full grid is the
   expected Chrome-crash path; do not attempt a full-size browser
   render as a pass criterion.
4. **Regression gate unchanged.** `node tests/harness/run.mjs smoke`
   (7 carts) must stay green - the new rules are inert in the calcite
   path, so smoke must be byte-identical aside from cabinet size.
   Spot-check one smoke cabinet's calcite render is unchanged.

## Out of scope

- Making Chrome actually survive the full cabinet (impossible by
  design - that's why calcite exists).
- Any change to the calcite engine.
- A build flag to suppress pixel rules (noted as future escape hatch
  only).
